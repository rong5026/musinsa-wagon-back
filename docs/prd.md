# Product Requirements Document - musinsa-wagon-nextjs-react-ts-v2

**Author:** Hong-yeonghwan
**Date:** 2025-12-06

## Executive Summary

**Musinsa Wagon**은 한국 주요 패션몰(무신사, 에이블리, 지그재그)의 **가격 투명성을 제공하는 웹 플랫폼**입니다.

### 핵심 가치 제안

20~30대 패션 소비자들이 명절 시즌마다 겪는 "가짜 할인" 속임수를 탐지하고, 브랜드별 상품 가격 히스토리를 추적하여 **"지금 사도 되는 가격인지"를 즉시 판단**할 수 있게 합니다.

**핵심 기능:**

- 📊 **브랜드 중심 가격 추적**: 쇼핑몰별 → 브랜드별 → 상품별 계층 구조로 가격 히스토리 제공
- 🚨 **명절 속임수 탐지**: 명절 전후 가격 조작 패턴 자동 감지
- ⚡ **실시간 가격 평가**: 30일/90일 최저가 대비 현재가 분석 (역대 최저가/좋은 딜/보통/비싼 편)
- ❤️ **찜 & 알림**: 사용자가 등록한 상품의 진짜 할인 시 알림

### What Makes This Special

**1. 패션 카테고리 특화**

- 기존 가격 추적 서비스(쿠차, keepa)는 전자제품 중심
- Musinsa Wagon은 한국 패션몰의 특수한 마케팅 패턴(명절 세일, 브랜드별 프로모션)을 이해하고 탐지

**2. 브랜드 중심 UX**

- 단순 상품 검색이 아닌, 쇼핑몰 → 브랜드 → 상품 구조로 사용자가 선호하는 브랜드를 쉽게 추적
- 멀티 플랫폼 통합 비교 (무신사/에이블리/지그재그)

**3. 실시간 판단 지원**

- 복잡한 그래프가 아닌, "지금 사세요" vs "조금 기다리세요" 같은 명확한 액션 가이드 제공

### 프로젝트 배경

개발자 포트폴리오이자 실제 20~30대 패션 소비자에게 실용 가치를 제공하는 서비스. MVP는 핵심 가격 추적 인프라와 명절 속임수 탐지 알고리즘 검증에 집중합니다.

## Project Classification

**Technical Type:** Web Application (SPA)
**Domain:** General Consumer Service
**Complexity:** Low

- **Tech Stack**: Next.js 15 + React 19 + TypeScript (Frontend) / Spring Boot API / Python 크롤러 / MySQL
- **Architecture**: 3-tier web application with scheduled crawling
- **Target Platform**: Modern browsers (Chrome, Safari, Edge)
- **Deployment**: Web-based (Vercel 또는 Docker)

## Product Scope

### MVP - Minimum Viable Product

**Core Features (Must Have)**

MVP에 반드시 포함되어야 하는 핵심 10개 기능:

**1. 사용자 인증 시스템**

- 회원가입 / 로그인 / 로그아웃
- 간단한 프로필 관리 (이메일, 닉네임)
- **이유**: 사용자별 찜 목록과 알림 설정 저장 필수

**2. 상품 등록 및 추적 (온디맨드 크롤링)**

- 사용자가 상품 URL을 입력하여 추적 등록
- 지원 플랫폼: 무신사, 에이블리, 지그재그
- 등록 즉시 크롤링 실행 및 현재 상품 정보 표시
- **이유**: 서비스의 진입점, 사용자가 직접 원하는 상품 선택

**3. 일일 배치 크롤링 시스템**

- 각 쇼핑몰 베스트 100개 + 카테고리별 50개 자동 크롤링
- 사용자가 등록한 모든 상품 일 1회 크롤링
- 가격 변동 히스토리 DB 저장
- **이유**: 가격 히스토리 데이터 축적 및 패턴 분석 기반

**4. 가격 히스토리 시각화**

- 상품 상세 페이지에서 30일/90일 가격 그래프 표시
- 최저가, 최고가, 평균가 표시
- **이유**: 사용자가 가격 변동을 직관적으로 이해

**5. 가격 평가 알고리즘 (Level 1)**

- 30일/90일 평균 대비 현재가 분석
- 4단계 라벨링: 🟢 역대 최저가 / 🟢 좋은 딜 / 🟡 보통 / 🔴 비싼 편
- 상품 리스트 및 상세 페이지에 라벨 표시
- **이유**: 핵심 가치 제안, 사용자가 즉시 판단 가능

**6. 명절 속임수 탐지 알고리즘 (Level 2)**

- 명절 전후 가격 패턴 분석 (D-14일 ~ D+7일)
- 원가 급등 후 할인 패턴 감지
- 🚨 "가짜 할인 의심" 경고 표시
- **이유**: 핵심 차별화 요소, MVP의 정체성

**7. 찜 목록 관리**

- 사용자별 찜 목록 조회
- 찜 목록에서 각 상품의 현재 가격 평가 상태 표시
- 찜 해제 기능
- **이유**: 사용자가 관심 상품을 한 곳에서 관리

**8. 목표가 설정 기능**

- 상품별로 목표가 설정 ("8만 원 이하 되면 알려줘")
- 찜 목록에서 목표가 대비 현재가 표시
- **이유**: 사용자 요청 반영, 구매 결정 명확화

**9. 푸시 알림 시스템**

- 역대 최저가 갱신 시 알림
- 사용자 설정 목표가 달성 시 알림
- 가짜 할인 탐지 시 경고 알림
- 알림 ON/OFF 설정 (상품별, 전체)
- **이유**: 핵심 가치 전달 수단, 사용자 재방문 유도

**10. 모바일 반응형 UI**

- 모바일 퍼스트 디자인
- 20~30대 친화적 모던 UI (Tailwind CSS + shadcn/ui)
- 색상 라벨로 즉시 인지 가능한 시각적 명확성
- **이유**: 타겟 사용자의 주 사용 환경

**MVP Success Criteria:**

- [ ] 명절 속임수 탐지 알고리즘 작동

- **이유**: 소셜 기능, MVP 이후

## User Journeys

### Journey 1: 지현 - 가짜 할인에서 벗어나기

지현은 28세 IT 기업 직장인으로, 출퇴근 지하철에서 무신사와 에이블리를 둘러보는 것이 일상이다. 이번 가을, 마음에 드는 코트를 무신사 장바구니에 담아두었다. 가격은 18만 원. "추석 세일 때 할인하면 그때 사야지"라고 생각했다.

추석 2주 전, 무신사 앱을 열었더니 그 코트가 "추석 대박 세일 50% 할인! 오늘만 9만 원!"이라고 떠 있었다. 지현은 "와, 진짜 반값이네!"하며 바로 결제했다. 뿌듯한 마음으로 친구 채팅방에 인증샷을 올렸다.

그런데 친구 수진이가 댓글을 달았다. "어? 나 그거 한 달 전에 10만 원에 샀는데?" 지현은 황당했다. 다시 확인해보니 "정상가 18만 원"이라는 표기만 있을 뿐, 평소 가격이 얼마였는지 알 방법이 없었다. **완전히 속은 기분이었다.**

다음 날, 지현은 "무신사 가격 추적"을 검색했고, 커뮤니티에서 Musinsa Wagon을 발견했다. 회원가입하고 앞으로 사고 싶은 상품 5개를 URL로 등록했다. 등록 즉시 각 상품의 30일 가격 그래프와 "🔴 비싼 편" 같은 라벨이 떴다. "아, 이거였구나. 이제 알겠어."

한 달 후, 푸시 알림이 왔다. "찜한 상품이 역대 최저가를 갱신했습니다! 🟢" 지현은 바로 Musinsa Wagon에 들어가 가격 그래프를 확인했다. 정말로 90일 동안 가장 낮은 가격이었다. 이번엔 안심하고 구매 버튼을 눌렀다.

6개월 후, 지현은 Musinsa Wagon 덕분에 평균 한 달에 10만 원 이상 절약하고 있다. 이제는 "명절 세일"이라는 말에 현혹되지 않는다. **똑똑한 소비자가 된 기분이다.**

**이 journey에서 드러나는 요구사항:**

- 상품 URL 등록 (온디맨드 크롤링)
- 30일/90일 가격 히스토리 그래프
- 4단계 가격 평가 라벨
- 찜 목록 관리
- 역대 최저가 푸시 알림
- 명절 전후 가격 패턴 분석

### Journey 2: 민수 - 개발자의 실용적 도구 발견

민수는 29세 프론트엔드 개발자로, 평소 GitHub에서 재미있는 오픈소스 프로젝트를 찾아보는 습관이 있다. 어느 날 트위터에서 "Next.js + Spring Boot + Python 크롤러로 만든 패션 가격 추적 서비스"라는 글을 보고 호기심이 생겼다.

"오, 풀스택 아키텍처네. 코드 한번 봐야지." 민수는 GitHub 리포지토리를 열었다. README를 읽다가 "실제 사용 가능한 서비스"라는 문구가 눈에 띄었다. "어차피 나도 에이블리에서 신발 몇 켤레 찜해뒀는데, 한번 써볼까?"

회원가입하고 찜해둔 신발 3켤레 URL을 등록했다. 등록 즉시 가격 그래프가 나타났다. "오, UI 깔끔한데? shadcn/ui 쓴 거 맞네." 민수는 개발자 눈으로 서비스를 살펴보면서도, 실제로 유용함을 느꼈다. "내가 찜한 이 신발이 지금 '보통' 가격이구나. 기다려볼까?"

일주일 후, "목표가 8만 원 달성!" 알림이 왔다. 민수는 즉시 Musinsa Wagon에 들어가 가격 그래프를 확인하고, 에이블리로 넘어가 구매했다. "이거 진짜 편한데? 가격 추적 자동화가 이렇게 유용할 줄이야."

며칠 후, 민수는 GitHub Issues에 UI 개선 제안을 남겼다. "찜 목록 정렬 기능 추가하면 좋을 것 같아요. 가격 낮은 순, 최저가 임박순 같은 거요." 그리고 개발자 커뮤니티 Slack에 "실용적인 포트폴리오 프로젝트 발견했다"며 링크를 공유했다.

민수는 이제 Musinsa Wagon의 얼리어답터이자 옹호자가 되었다. **기술도 보고 실용성도 얻는 일석이조.**

**이 journey에서 드러나는 요구사항:**

- GitHub 오픈소스 접근성
- 직관적인 UI/UX (shadcn/ui)
- 목표가 설정 및 알림
- 찜 목록 관리
- 피드백 제공 채널 (GitHub Issues)
- 개발자 커뮤니티 공유 가능성

### Journey 3: 지현 (다시) - 명절 속임수 탐지 경험

이번엔 지현이 더 발전된 시나리오다. 설날이 다가오고 있고, 지현의 찜 목록에는 10개 상품이 있다. 설날 2주 전부터 Musinsa Wagon이 이상한 패턴을 감지했다.

어느 날 아침, 푸시 알림이 왔다. "🚨 가짜 할인 의심: 찜한 상품의 정가가 급등했습니다. 명절 전후 가격 조작 패턴이 감지되었습니다."

지현은 Musinsa Wagon에 들어갔다. 찜한 니트 가격이 12만 원에서 18만 원으로 올랐다가, "설날 대박 세일 50% 할인! 9만 원!"으로 표시되어 있었다. 하지만 Musinsa Wagon의 그래프를 보니, 평소 가격은 12만 원이었고, 14일 전부터 인위적으로 올라간 게 분명했다.

지현은 웃으며 생각했다. "작년 추석 때 당했던 거랑 똑같네. 이번엔 안 속아." 그녀는 그 상품을 구매하지 않고, 대신 Musinsa Wagon이 "🟢 역대 최저가"로 표시한 다른 상품을 샀다.

설날이 지나고 1주일 후, 그 니트 가격은 다시 12만 원으로 돌아왔다. 지현은 친구들 단톡방에 스크린샷을 공유했다. "봐봐, Musinsa Wagon이 가짜 할인 잡아줬어. 진짜 신기하지 않아?" 친구 3명이 바로 Musinsa Wagon에 가입했다.

**지현은 이제 Musinsa Wagon 없이는 명절 쇼핑을 할 수 없다.**

**이 journey에서 드러나는 요구사항:**

- 명절 속임수 탐지 알고리즘 (D-14일 ~ D+7일)
- 가격 급등 패턴 감지
- 가짜 할인 경고 푸시 알림
- 가격 히스토리 그래프 (명절 전후 비교)
- SNS 공유 가능성 (바이럴)

### Responsive Design Strategy

**모바일 퍼스트 접근:**

**브레이크포인트:**

- **Mobile**: 320px ~ 767px (주 타겟)
- **Tablet**: 768px ~ 1023px (부차적)
- **Desktop**: 1024px 이상 (선택적)

**디자인 우선순위:**

1. **Mobile (최우선)**: 전체 UI/UX 설계의 80% 집중

   - 출퇴근 지하철에서 사용하는 시나리오
   - 한 손으로 조작 가능한 인터페이스
   - 최소한의 스크롤로 핵심 정보 표시

2. **Tablet (중간)**: Mobile 레이아웃 확장

   - 카드 그리드 2열 표시
   - 가격 그래프 더 크게 표시

3. **Desktop (선택적)**: 추가 정보 표시
   - 사이드바 네비게이션
   - 멀티 컬럼 레이아웃
   - 더 상세한 가격 히스토리 그래프

**핵심 컴포넌트 반응형 요구사항:**

- 찜 목록: Mobile 1열 → Tablet 2열 → Desktop 3열
- 가격 그래프: Mobile 세로 스크롤 → Desktop 가로 확장
- 상품 카드: Touch-friendly 터치 영역 (최소 44x44px)

**번들 크기:**

- Initial JS bundle: < 200KB (gzipped)
- Total JS (initial + chunks): < 500KB
- CSS: < 50KB (Tailwind CSS purge)

### Accessibility Requirements

**접근성 수준: AA (WCAG 2.1 기준)**

**근거:**

- 법적 요구사항은 없지만, 모든 사용자에게 공평한 경험 제공
- 색각 이상자도 가격 평가 라벨 구분 가능해야 함
- 스크린 리더 사용자도 기본 기능 사용 가능

**핵심 요구사항:**

1. **색상 대비**

   - 텍스트-배경 대비: 최소 4.5:1 (WCAG AA)
   - 가격 라벨 색상: 색상만이 아닌 아이콘/텍스트 병행
     - 🟢 역대 최저가 → 초록색 + "역대 최저가" 텍스트
     - 🔴 비싼 편 → 빨간색 + "비싼 편" 텍스트

2. **키보드 네비게이션**

   - 모든 인터랙티브 요소 Tab 키로 접근 가능
   - Focus 상태 명확히 표시 (outline)
   - 논리적 탭 순서

3. **스크린 리더 지원**

   - `<button>`, `<a>` 등 시맨틱 요소 사용
   - `aria-label` 적절히 활용
   - 상품 카드: "상품명, 현재가, 가격 평가"를 순서대로 읽음

4. **반응형 폰트 크기**

   - 최소 16px (모바일)
   - 사용자 브라우저 폰트 크기 설정 존중 (rem 단위)

5. **에러 메시지 및 알림**
   - 명확한 에러 메시지
   - aria-live 영역으로 스크린 리더에 알림

**비요구사항 (MVP 범위 외):**

- AAA 수준 접근성 (높은 대비 모드)
- 다국어 지원
- 수어 영상

### Real-Time Features

**실시간 기능 요구사항:**

**1. 푸시 알림 시스템**

- **기술**: Firebase Cloud Messaging (FCM) 또는 Web Push API

  - 역대 최저가 갱신

**3. WebSocket/SSE 불필요**

- 가격 변동은 일 1회이므로 실시간 연결 불필요
- 알림은 푸시 메시지로 충분

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach: Problem-Solving MVP**

Musinsa Wagon의 MVP는 "가짜 할인 탐지"라는 핵심 문제를 해결하는 것에 집중합니다. 사용자가 명절 시즌에 겪는 가격 속임수를 탐지하고, 진짜 할인 시점을 알려주는 것이 핵심 가치입니다.

**Resource Requirements:**

- **개발자**: 1명 (풀스택)
- **투입 시간**: 주 10시간 (파트타임)
- **개발 기간**: 18~22주 (약 5~6개월)
- **기술 스택**: Next.js 15 + React 19 + TypeScript / Spring Boot / Python 크롤러 / MySQL
- **인프라**: Vercel (Frontend), AWS/GCP (Backend), MySQL DB

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**

MVP는 3가지 핵심 사용자 여정을 모두 지원합니다:

1. **지현의 가짜 할인 탐지 여정**

   - 상품 URL 등록 → 가격 히스토리 확인 → 가격 평가 라벨 확인 → 진짜 할인 시 알림 수신

2. **민수의 개발자 실용 도구 여정**

   - GitHub 발견 → 회원가입 → 찜 상품 등록 → 목표가 알림 수신 → 피드백 제공

3. **지현의 명절 속임수 탐지 여정**
   - 명절 전 가격 급등 감지 → 가짜 할인 경고 수신 → 구매 보류 → 친구에게 공유

**Must-Have Capabilities (10개 핵심 기능):**

1. **사용자 인증 시스템**

   - 회원가입 / 로그인 / 로그아웃
   - 간단한 프로필 관리 (이메일, 닉네임)

2. **상품 등록 및 추적 (온디맨드 크롤링)**

   - 사용자가 상품 URL을 입력하여 추적 등록
   - 지원 플랫폼: 무신사, 에이블리, 지그재그
   - 등록 즉시 크롤링 실행 및 현재 상품 정보 표시

3. **일일 배치 크롤링 시스템**

   - 각 쇼핑몰 베스트 100개 + 카테고리별 50개 자동 크롤링
   - 사용자가 등록한 모든 상품 일 1회 크롤링
   - 가격 변동 히스토리 DB 저장

4. **가격 히스토리 시각화**

   - 상품 상세 페이지에서 30일/90일 가격 그래프 표시
   - 최저가, 최고가, 평균가 표시

5. **가격 평가 알고리즘 (Level 1)**

   - 30일/90일 평균 대비 현재가 분석
   - 4단계 라벨링: 🟢 역대 최저가 / 🟢 좋은 딜 / 🟡 보통 / 🔴 비싼 편
   - 상품 리스트 및 상세 페이지에 라벨 표시

6. **명절 속임수 탐지 알고리즘 (Level 2)**

   - 명절 전후 가격 패턴 분석 (D-14일 ~ D+7일)
   - 원가 급등 후 할인 패턴 감지
   - 🚨 "가짜 할인 의심" 경고 표시

7. **찜 목록 관리**

   - 사용자별 찜 목록 조회
   - 찜 목록에서 각 상품의 현재 가격 평가 상태 표시
   - 찜 해제 기능

8. **목표가 설정 기능**

   - 상품별로 목표가 설정 ("8만 원 이하 되면 알려줘")
   - 찜 목록에서 목표가 대비 현재가 표시

9. **푸시 알림 시스템**

   - 역대 최저가 갱신 시 알림
   - 사용자 설정 목표가 달성 시 알림
   - 가짜 할인 탐지 시 경고 알림
   - 알림 ON/OFF 설정 (상품별, 전체)

10. **모바일 반응형 UI**
    - 모바일 퍼스트 디자인
    - 20~30대 친화적 모던 UI (Tailwind CSS + shadcn/ui)
    - 색상 라벨로 즉시 인지 가능한 시각적 명확성

**MVP Success Criteria:**

기능 체크리스트:

- [ ] 무신사, 에이블리, 지그재그 URL로 상품 등록 가능
- [ ] 등록 즉시 상품 정보 크롤링 및 표시
- [ ] 일일 배치 크롤링 자동 실행 (95% 성공률)
- [ ] 가격 히스토리 그래프 표시 (30일/90일)
- [ ] 4단계 가격 평가 라벨 표시
- [ ] 명절 속임수 탐지 알고리즘 작동
- [ ] 찜 목록 조회 및 관리
- [ ] 목표가 설정 및 표시
- [ ] 3가지 푸시 알림 작동 (최저가/목표가/가짜할인)
- [ ] 모바일 반응형 UI 완성

기술적 기준:

- [ ] 회원가입~알림까지 전체 플로우 end-to-end 작동
- [ ] 최소 3명의 테스터가 1주일 사용 가능 (크래시 없이)
- [ ] 실제 도메인에 배포 완료
- [ ] 크롤링 실패 시 에러 핸들링 및 로깅

데이터 기준:

- [ ] MVP 출시 전 1개월간 인기 상품 히스토리 수집 완료
- [ ] 최소 300개 상품의 가격 히스토리 보유

### Post-MVP Features (Phase 2 - Growth)

MVP에서는 제외하고 향후 개발할 기능들:

**브라우저 확장 프로그램**

- 현재: URL 복붙으로 등록
- 나중에: 쇼핑몰에서 클릭 한 번에 등록
- **이유**: 추가 개발 시간 필요, 코어 기능 아님

**커뮤니티 기능**

- 사용자들끼리 "진짜 할인" 정보 공유
- 댓글, 좋아요 등
- **이유**: MVP는 개인 도구에 집중, 소셜 기능은 사용자 확보 후

**브랜드/카테고리 분석**

- "이 브랜드는 보통 언제 할인해요"
- "아우터는 2월에 가장 싸요"
- **이유**: 충분한 히스토리 데이터 필요 (6개월 이상)

**추가 쇼핑몰**

- 29CM, W컨셉, SSF샵 등
- **이유**: 크롤러 개발 시간, 일단 3개로 검증

**이메일 알림**

- 푸시 알림만 우선
- **이유**: 푸시가 더 즉시성 있고, 이메일은 추가 인프라 필요

**가격 예측**

- "다음 주에 더 싸질 것 같아요"
- **이유**: AI/ML 필요, MVP 범위 초과

**찜 공유**

- "내 찜 목록 친구에게 공유"
- **이유**: 소셜 기능, MVP 이후

### Phase 3 (Expansion - Future Vision)

**장기 비전:**

- 한국 패션 가격 투명성의 표준 플랫폼
- 전자제품을 넘어 패션 카테고리의 가격 추적 리더
- 사용자 커뮤니티가 만드는 "진짜 할인" 정보 허브
- 브랜드와 쇼핑몰에게도 가치 제공 (정당한 할인 마케팅 인증)

**확장 가능성:**

- 추가 카테고리 확장 (뷰티, 라이프스타일)
- 해외 쇼핑몰 지원 (ASOS, Zara 등)
- B2B 서비스 (브랜드/쇼핑몰 대상 가격 최적화 컨설팅)
- 모바일 앱 (iOS, Android)

### Development Timeline & Priorities

**Phase 1: 인프라 (4~5주)**

1. 사용자 인증 시스템 (Spring Security + JWT)
2. DB 스키마 설계 (Users, Products, PriceHistory, Watchlist, Notifications)
3. Python 크롤러 ↔ Spring Boot 연동
4. 일일 배치 크롤링 자동화 (Spring Scheduler)

**Phase 2: 코어 기능 (6~7주)** 5. 상품 등록 API 및 UI 6. 가격 히스토리 조회 API 7. 가격 평가 알고리즘 (Level 1) 8. 찜 목록 CRUD 9. 상품 상세 페이지 UI (그래프 포함 - Chart.js 등)

**Phase 3: 차별화 기능 (4~5주)** 10. 명절 속임수 탐지 알고리즘 (Level 2) 11. 목표가 설정 기능 12. 푸시 알림 시스템 (Firebase Cloud Messaging)

**Phase 4: 폴리싱 (4~5주)** 13. 모바일 UI 최적화 14. 에러 핸들링 및 사용자 피드백 메시지 15. 성능 최적화 (페이지 로딩 속도, 쿼리 최적화) 16. 테스트 및 버그 수정

**Phase 5: 사전 데이터 수집 (4주, Phase 1~4와 병행 가능)** 17. 베스트/카테고리 상품 크롤링 시작 (Python 크롤러 24시간 작동) 18. 1개월간 히스토리 축적 (백그라운드 작업)

**총 예상 기간: 18~22주 (약 5~6개월)** (주 10시간 파트타임 기준)

**마일스톤:**

- **M1 (5주차)**: 인프라 완성, 수동 크롤링 가능
- **M2 (12주차)**: 기본 가격 추적 기능 작동, 알파 테스트 시작
- **M3 (17주차)**: 명절 탐지 알고리즘 완성, 베타 테스트
- **M4 (22주차)**: MVP 완성, 공개 론칭

### Risk Mitigation Strategy

**Technical Risks:**

1. **크롤링 안정성**

   - **Risk**: 쇼핑몰 사이트 구조 변경으로 크롤러 실패
   - **Mitigation**:
     - 에러 핸들링 및 로깅 철저히 구현
     - 크롤러 실패 시 자동 재시도 (3회)
     - 크롤러 실패율 모니터링 대시보드
     - 쇼핑몰별 백업 크롤링 전략 (API vs Selenium)

2. **데이터 히스토리 부족**

   - **Risk**: MVP 출시 시 가격 히스토리 데이터가 부족하여 알고리즘이 정확하지 않음
   - **Mitigation**:
     - MVP 출시 1개월 전부터 사전 데이터 수집 시작
     - 최소 300개 인기 상품의 30일 히스토리 확보
     - 히스토리가 부족한 상품은 "데이터 수집 중" 표시

3. **푸시 알림 인프라**
   - **Risk**: FCM 설정 복잡성, 푸시 알림 미도착
   - **Mitigation**:
     - MVP에서는 Web Push API로 시작 (FCM은 Post-MVP)
     - 알림 전송 로그 및 실패율 모니터링
     - 사용자에게 알림 설정 가이드 제공

**Market Risks:**

1. **사용자 확보 실패**

   - **Risk**: 초기 100명 사용자 확보 목표 미달
   - **Validation Approach**:
     - 개발자 커뮤니티 우선 공개 (Reddit, 개발자 카페)
     - GitHub README에 명확한 가치 제안
     - 친구/지인 베타 테스터 최소 10명 확보
     - 실제 절약 사례를 SNS에 공유

2. **사용자 이탈**
   - **Risk**: 초기 사용자가 재방문하지 않음
   - **Validation Approach**:
     - 푸시 알림으로 재방문 유도
     - 찜 목록 확인 주간 리마인더
     - 사용자 피드백 수집 및 빠른 개선

**Resource Risks:**

1. **개발 시간 부족**

   - **Risk**: 주 10시간 투입이 어려워 일정 지연
   - **Contingency**:
     - MVP 기능 우선순위 재조정 (명절 탐지 알고리즘을 Post-MVP로 이동 가능)
     - 가장 핵심적인 5개 기능만 먼저 구현
     - 일정 유연성 확보 (22주 → 30주까지 허용)

2. **혼자 개발하는 한계**
   - **Risk**: 풀스택 개발의 복잡도, 버그 발견 어려움
   - **Contingency**:
     - 코드 리뷰를 개발자 커뮤니티에 요청
     - 베타 테스터를 통한 조기 버그 발견
     - 기술 스택 단순화 (복잡한 기능 제거)
